## 1. 项目背景与目标

构建一个安卓充电日志分析系统，核心功能是从日志中提取 `healthd` 关键字数据进行解析、分析、可视化。

- **核心计算**：C++17（编译为独立可执行文件，Python 通过 subprocess 调用）
- **Web API**：Flask (Python)，环境已有 Python3 + Flask 3.1.3
- **前端**：前后端分离，Vue 3 + TypeScript + Vite + ECharts
- **输入**：CLI 命令行 + Web 页面上传，两者调用同一个 C++ 可执行文件
- **扩展性**：解析器使用类继承设计，未来可适配不同的充电日志格式
- **存储**：SQLite（轻量、单文件、零运维）
- **构建**：CMake（C++ 部分）
- **仓库**：Monorepo

## 4. C++ Core 库设计

### 4.1 解析器体系 (可扩展的类继承)

```cpp
// base_charger_parser.h - 所有日志解析器的基类
class BaseParser {
public:
    virtual ~BaseParser() = default;
    // 判断本解析器能否处理该行
    virtual bool canParse(const std::string& line) const = 0;
    // 解析一行，返回 DataPoint 或 nullopt
    virtual std::optional<ChargerDataPoint> parseLine(const std::string& line) const = 0;
    // 解析器名称标识
    virtual std::string parserName() const = 0;
};

// healthd_parser.h - Android healthd 日志解析
class HealthdParser : public BaseParser { ... };
```

> **当前状态**: 仅实现了 `HealthdParser`，CLI 中直接实例化。`ParserFactory` 工厂模式为未来扩展预留，尚未实现。

### 4.2 Healthd 日志格式

实际 Android healthd 行示例：
```
healthd: battery l=100 v=4099 t=31.2 h=2 st=3 c=-889500 fc=2946000 cc=5 chg=
```

字段映射：

| 字段 | 含义 | 单位 | 映射 |
|------|------|------|------|
| `l` | 电池电量 | % | `battery_level_pct` |
| `v` | 电池电压 | mV | `battery_voltage_mv` |
| `t` | 电池温度 | °C | `battery_temperature_c` |
| `c` | 电池电流 | µA | `battery_current_ma` (÷1000 转 mA) |
| `h` | 电池健康 | enum | (未解析: 2=正常) |
| `st` | 充电状态 | enum | (未解析: 2=充电, 3=放电, 5=满) |
| `fc` | 满电容量 | — | (未解析) |
| `cc` | 充电循环次数 | count | `charge_cycle_count` |
| `chg` | 充电器类型 | char | (未解析: 空/a=AC) |

> **canParse 过滤**: 仅匹配同时包含 `healthd` 和 `battery` 的行，排除 ramoops 等文件中的非电池日志（如 `healthd peak:100, real_cap:80...`）。  
> **时间戳校验**: `parseLine` 若无法从行首提取 `MM-DD HH:MM:SS.mmm` 格式时间戳，返回 `nullopt` 跳过该行，避免产出 `t=0` 的无效数据点。

### 4.3 数据模型

```cpp
struct ChargerDataPoint {
    int64_t elapsed_ms = 0;            // 年内毫秒数: 自 Jan 1 00:00:00.000 起 (解析 MM-DD HH:MM:SS.mmm)
    std::string timestamp_str;         // 原始时间戳字符串 (MM-DD HH:MM:SS.mmm, 不含年份)

    double battery_current_ma = NAN;       // 电池电流 (mA, 由 c 字段 µA÷1000)
    double battery_voltage_mv = NAN;       // 电池电压 (mV)
    double bus_voltage_mv = NAN;           // VBUS 线上电压 (mV)
    double bus_current_ma = NAN;           // 线上电流 (mA)
    double battery_temperature_c = NAN;    // 电池温度 (°C)
    double battery_level_pct = NAN;        // 电池电量百分比 (%)
    double charge_cycle_count = NAN;       // 充电循环次数 (cc)

    bool has(const std::string& field) const;
    double get(const std::string& field) const;
    std::vector<std::string> availableFields() const;
};

// 阈值分析数据结构
struct ThresholdSegment {
    int64_t start_ms = 0;
    int64_t end_ms = 0;
};

struct ThresholdResult {
    std::string field_name;
    double threshold_value = NAN;
    int64_t total_time_ms = 0;
    int64_t above_time_ms = 0;
    double above_pct = 0.0;
    std::vector<ThresholdSegment> above_segments;
};

```

### 4.4 统计计算 & 缓存

```cpp
class StatsCalculator {
public:
    // 单字段统计 (min/max/avg/count)
    static FieldStats calcField(const std::vector<ChargerDataPoint>& points,
                                const std::string& field,
                                int64_t start_ms = 0, int64_t end_ms = INT64_MAX);
    // 全字段批量统计
    static std::vector<FieldStats> calcAllFields(
        const std::vector<ChargerDataPoint>& points,
        const std::vector<std::string>& fields);
    // 降采样 (均匀间距保留首尾点)
    static std::vector<ChargerDataPoint> downsample(
        const std::vector<ChargerDataPoint>& points, size_t target_count);
    // 阈值分析: 线性插值计算某字段超过阈值的累计时间和时段
    static ThresholdResult calcThreshold(
        const std::vector<ChargerDataPoint>& points,
        const std::string& field, double threshold,
        int64_t start_ms = 0, int64_t end_ms = INT64_MAX);
};

class CacheManager {
public:
    // FNV-1a 64-bit 目录指纹 (文件列表 + mtime)
    static uint64_t computeFingerprint(const std::filesystem::path& dir);
    // 保存缓存 (pipe分隔文本, 原子写入 .tmp → rename)
    static bool save(const std::filesystem::path& cache_path,
                     uint64_t fingerprint,
                     const std::vector<ChargerDataPoint>& points);
    // 加载缓存 (指纹不匹配或格式损坏返回 nullopt)
    static std::optional<std::vector<ChargerDataPoint>>
        load(const std::filesystem::path& cache_path, uint64_t fingerprint);
};
```

### 4.5 CLI 接口 (Python 通过 subprocess 调用)

```bash
# 单命令模式：解析日志目录，输出 JSON 到 stdout
chargerlog [--json] [--points] [--downsample N] [--no-cache]
           [--start HH:MM:SS] [--end HH:MM:SS]
           [--threshold field=value]... <log_directory>

# 基本解析（人类可读输出）
chargerlog /path/to/logs

# JSON 输出（供 Python cpp_bridge 解析）
chargerlog --json /path/to/logs

# 含降采样数据点
chargerlog --json --points --downsample 500 /path/to/logs

# 跳过缓存强制重新扫描
chargerlog --json --no-cache /path/to/logs

# 时间范围过滤 + 阈值分析
chargerlog --json --start 09:00:00 --end 18:00:00 \
    --threshold battery_voltage_mv=4200 \
    --threshold battery_temperature_c=40 \
    /path/to/logs
# JSON 输出含 "stats" 和 "thresholds" 数组
```

> `--json` 输出为标准 JSON，包含统计数据和时间范围内的阈值分析结果。Python 端通过 `subprocess.run()` 调用，解析 stdout JSON。不再有 `parse/import/list/show/chart` 子命令 —— 所有功能通过单命令 + 标志完成。

## 5. Flask API 设计

Python 通过子进程调用 C++ 可执行文件，stdin/stdout 传递 JSON 数据。

### cpp_bridge.py

```python
import subprocess
import json

CHARGERLOG_BIN = "chargerlog"  # 或配置路径

def run(*args) -> dict:
    """调用 chargerlog，返回解析后的 JSON"""
    result = subprocess.run(
        [CHARGERLOG_BIN] + list(args),
        capture_output=True, text=True, timeout=60
    )
    if result.returncode != 0:
        raise ChargerLogError(result.stderr)
    return json.loads(result.stdout)
```

### API 路由

| Method | Path | 说明 |
|--------|------|------|
| POST | `/api/upload` | 分析日志目录或压缩包，调用 `chargerlog` 返回 `{ analysis_id, points, fields }` |
| POST | `/api/select-path` | 打开原生文件/目录选择对话框 (tkinter)，返回 `{ path }` |
| GET | `/api/analysis/:id` | 获取分析结果。每次重新调用 chargerlog，利用 `.chargerlog_cache` 判断是否需要重新扫描 |
| GET | `/api/history` | 历史分析列表，支持 `?page=&limit=` 分页 |
| GET | `/api/history/:id` | 单条历史分析详情 |
| DELETE | `/api/history/:id` | 删除分析记录 |

> `/api/upload` 支持目录路径和压缩文件路径 (`.zip`, `.tar.gz`, `.tar.bz2`, `.tar.xz`, `.tgz`, `.tar`)，压缩文件自动调用 `decompressor.py` 解压后再分析。不再支持 `start`/`end` 参数——时间范围筛选已移至前端 Dashboard 的时间范围滑块。

## 6. 存储设计

> **当前方案**: SQLite 方案未实现。实际采用双轨文件存储：

### 6.1 C++ 端：文本缓存

`.chargerlog_cache` 文件存放在被扫描的日志目录中，pipe (`|`) 分隔的纯文本格式：

- **Line 1**: `V1|<fingerprint_16_hex>`
- **数据行**: `elapsed_ms|voltage_mv|temp_c|current_ma|level_pct|bus_voltage_mv|bus_current_ma|cc|timestamp_str`
- NaN 值序列化为 `"nan"`

指纹通过 FNV-1a 64-bit 对目录中所有文件的 `(相对路径, mtime, 大小)` 排序后计算。指纹不匹配时缓存失效，自动重新扫描。写入采用原子操作 (`.tmp` → `rename`)。

### 6.2 Python 端：JSON 文件

`server/history/<analysis_id>.json` — 每个分析一个 JSON 文件，存储完整分析结果（含降采样后的数据点），供 Dashboard 加载和历史查询。

### 6.3 未来规划：SQLite

以下 SQL schema 为未来大数据量场景预留，当前未实现：

CREATE TABLE analyses (

## 7. 前端设计

技术栈：Vue 3 + TypeScript + Vite + ECharts（状态管理使用 Composition API `ref()`/`computed()`，未使用 Pinia）

### 页面路由

| 路由 | 页面 | 说明 |
|------|------|------|
| `/upload` | UploadPage | 新建分析：输入日志路径 (支持目录/压缩包) + 原生文件浏览器按钮 |
| `/dashboard/:analysisId` | Dashboard | 时间范围滑块 + 4 个可拖拽排序图表 + 阈值分析 |
| `/history` | HistoryPage | 历史分析列表，可排序、搜索 |

### UploadPage 新建分析

- 日志路径输入框支持**目录路径**和**压缩文件路径** (`.zip`, `.tar.gz` 等)
- 输入框右侧"浏览"按钮 → 调用 `POST /api/select-path` → 服务端 tkinter 打开 Windows 原生文件管理器 → 返回路径填入
- 压缩文件由服务端 `decompressor.py` 自动解压后再分析
- **不再显示**起始/结束时间输入框——时间范围在 Dashboard 上基于实际数据选择

### Dashboard 图表

| # | 图表 | 类型 | 说明 |
|---|------|------|------|
| 1 | 电压 + 温度 | 双 Y 轴组合图 | 左轴 mV / 右轴 °C，Legend 可切换可见性 |
| 2 | 电流 | 独立折线图 | mA，有正负方向 |
| 3 | 电量 | 独立折线图 | %，Y 轴 0~100 |
| 4 | 剩余循环次数 | 独立折线图 | `额定次数 - cc`，额定值可在图表右上角编辑 |

- **时间范围滑块**：Dashboard 顶部显示数据全量时间范围，双 handle 可拖拽滑块 + MM-DD HH:MM:SS 精确输入框，支持"重置全部参数"恢复全量并清空所有阈值。异常时间 (负数、> 366 天) 自动过滤。所有图表和阈值统计均基于选定时间范围实时更新
- 图表卡片支持**拖拽排序**（HTML5 Drag & Drop），排序后 ECharts 实例自动跟随 DOM 移动
- ECharts 原生 `init()`/`dispose()` 管理生命周期，禁用内置 ResizeObserver（避免 minimize 弹回 bug）
- 每个图表支持缩放、tooltip、时间轴联动
- **阈值分析**：每个图表可设置一个阈值，低于阈值的折线段以 ~25% 不透明度**半透明**渲染，≥阈值的折线段**加粗**（width: 4）渲染。前端通过 `threshold.ts` 纯函数从降采样点计算超阈值时间及占比，与 C++ `calcThreshold()` 使用相同的线性插值算法
- 阈值输入框使用 `:value` + `@input` 手动绑定，清空时写入 `null` 关闭阈值效果（避免 `v-model.number` 把空字符串变为 `0` 的 bug）
- Dashboard 拆分为独立组件：`TimeRangeSlider`（时间范围滑块）、`StatsTable`（汇总表）、`ChartCard`（图表卡片容器 + 拖拽），Dashboard.vue 负责数据加载与图表编排

## 10. PyInstaller 打包（Windows 桌面应用）

用户无需安装 Python、Node.js 或任何依赖即可运行。双击 exe → Flask 自动启动 → 浏览器打开 → 完成分析。

### 10.1 打包方案

PyInstaller `--onedir` 模式（COLLECT 输出），生成完整可分发目录。

### 10.2 输出结构

```
dist/ChargerLogApp/
├── ChargerLogApp.exe         ← 双击运行（无控制台窗口）
├── history/                  ← 分析记录持久化（首次运行时自动创建）
└── _internal/                ← Python 运行时 + 所有依赖
    ├── chargerlog.exe        ← C++ 日志解析引擎
    ├── web/                  ← Vue 前端静态文件
    │   ├── index.html

    ├── *.dll                 ← Windows 系统 DLL
    └── _tcl_data/, _tk_data/ ← tkinter 文件对话框所需资源
```

### 10.3 路径解析

`server/config.py` 集中管理所有路径，三层优先级：

| 优先级 | 机制 | 说明 |
|--------|------|------|
| 1 | 环境变量 | `CHARGERLOG_BIN_PATH`、`CHARGERLOG_HISTORY_DIR` 可覆盖 |
| 2 | `sys.frozen` 检测 | PyInstaller 打包后 → `sys._MEIPASS` / `sys.executable.parent` |
| 3 | `__file__` 相对路径 | dev 模式 → 项目目录结构 |

- **chargerlog.exe**: frozen → `sys._MEIPASS/chargerlog.exe`；dev → `core/build/chargerlog.exe`
- **前端静态文件**: frozen → `sys._MEIPASS/web/`；dev → `web/dist/`
- **历史记录**: frozen → `sys.executable.parent/history/`（exe 所在目录，持久化）；dev → `server/history/`

> `launcher.py` 启动时通过环境变量注入路径，`config.py` 优先读取环境变量。`upload.py` 中 history 目录通过 `_get_history_dir()` 动态获取（非模块级缓存），确保 frozen/dev 模式切换正确。

### 10.4 无控制台 + 自动关闭

- **`chargerlog.spec`**: `console=False` → PyInstaller 使用 `runw.exe` 引导程序（Windows GUI 子系统），双击不弹出黑色控制台
- **心跳监控**: 前端每 10 秒 `POST /api/heartbeat` → 服务端更新时间戳；若 120 秒无心跳则 `os._exit(0)` 自动退出
- **页面关闭**: 前端 `beforeunload` 时 `navigator.sendBeacon('/api/shutdown')` → 服务端立即退出
- **心跳函数注入**: 通过 `app.config['_heartbeat_update']` 注入回调，**禁止在 API 层 `import launcher`**——PyInstaller 会将入口脚本二次加载为独立模块，导致 `__main__` 和 `launcher` 各持一份变量副本，心跳监控失效

### 10.5 子进程黑窗消除

`cpp_bridge.py` 调用 `subprocess.run()` 时传入 `creationflags=subprocess.CREATE_NO_WINDOW`（Windows），避免 `chargerlog.exe` 子进程弹出 cmd 黑窗。

### 10.6 构建脚本

根目录下 `build.py` — 单文件跨平台打包脚本：

```bash
python build.py                    # 完整构建
python build.py --clean            # 清理后重新构建
python build.py --skip-cpp         # 跳过 C++ 编译（只改了前端/Python 时用）
python build.py --skip-web         # 跳过前端构建（只改了 C++ 时用）
```

流程：C++ 编译 (`cmake --build`) → 前端构建 (`npm run build`) → PyInstaller 打包 → 清理中间 EXE。  
`shell=True` 确保 Windows 上能找到 `.cmd`/`.bat` 文件（如 `npm.cmd`）。

### 10.8 已知陷阱

| 陷阱 | 现象 | 解决 |
|------|------|------|
| `pathex` 为空 | `ModuleNotFoundError: No module named 'app'` | 添加 `str(ROOT / "server")` 到 pathex |
| API 层 `import launcher` | 页面卡死/历史空白，~30s 后进程被杀 | 通过 `app.config` 注入回调，禁止跨层 import |
| `console=True` | 双击弹出黑色控制台 | `console=False` |
| 子进程无 `CREATE_NO_WINDOW` | 分析时闪过 cmd 黑窗 | `creationflags=subprocess.CREATE_NO_WINDOW` |
| `HISTORY_DIR` 模块级缓存 | 移动到其他目录后 history 写回项目目录 | 改为 `_get_history_dir()` 每次动态调用 |

### 10.9 文件对话框 DPI 感知

`tkinter.filedialog` 在 Windows 高 DPI 显示器下默认模糊。在 `upload.py` 模块顶部调用 Windows API 启用 Per-Monitor DPI 感知：

```python
# server/routes/upload.py
if os.name == "nt":
    try:
        ctypes.windll.shcore.SetProcessDpiAwareness(2)
    except Exception:
        try:
            ctypes.windll.user32.SetProcessDPIAware()
        except Exception:
            pass
```

必须在 `tk.Tk()` 创建任何窗口之前调用。回退链路覆盖新旧 Windows API。

### 10.10 静态文件服务（避免 Jinja2 TemplateNotFound）

Flask 内置 `static_url_path=''`（根路径静态文件）会在不同 Flask/Werkzeug 版本或平台上产生不同的路由行为，表现为 `jinja2.exceptions.TemplateNotFound: index.html`。

改为**显式路由**：Flask 构造函数不传 `static_folder` 参数，改用 `@app.route("/<path:filename>")` 手动调用 `send_from_directory` 提供前端文件。Flask 内置 static handler 完全被旁路，消除了版本/平台差异。

```python
# server/app.py — create_app()
app = Flask(__name__)  # 不传 static_folder

# ...
if static_folder and Path(static_folder).is_dir():
    @app.route("/")
    @app.route("/<path:filename>")
    def serve_frontend(filename: str = "index.html"):
        return send_from_directory(static_folder, filename)
else:
    @app.route("/")
    def index():
        return r"""..."""  # 内联降级页面
```

> `launcher.py` 同时写入 `launcher.log`（仅 frozen 模式），记录 static/chargerlog/history 路径及文件列表，便于排查部署问题。

### 10.11 Dashboard 图表统计数据

每个图表卡片标题栏居中显示当前时间范围内的统计数据，同时提供汇总表。

**汇总表**：可拖拽排序的统计卡片（`chartOrder` 中 type 为 `stats`），以表格形式展示所有指标的最大值 / 最小值 / 平均值，每行带对应曲线颜色圆点。

**内联统计**：每个图表标题栏中央，以紧凑上标格式显示 max/min/avg：

```
电池电流 1200^max -1174^min -120^avg mA
```

- `max`/`min`/`avg` 标签为 8px 上标（`vertical-align: super`），紧贴数值右侧
- 布局使用 CSS grid 三列（`auto 1fr auto`），标题左对齐、统计居中、阈值控件右对齐，互不重叠
